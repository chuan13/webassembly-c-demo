C 側底層：`uint8_t` 到 `Pixel` 轉型原理

記憶體佈局的契合

  • 連續一維陣列：HTML5 Canvas 輸出的像素是 [R,G,B,A, R,G,B,A, ...]，資料型態為 uint8。
  • 零 Padding 結構體：在 C 中定義 Pixel 包含四個 uint8_t，總大小精確為 4 bytes。編譯器不會在此結構中插入任何填充空間。
  • 指標算術 (Pointer Arithmetic)：強轉型 Pixel* img = (Pixel*)ptr; 後，img[i] 會自動以 4 bytes 為單位進行偏移，達到直觀、安全的快速像素操作。
// 像素結構體與應用範例
typedef struct {
    uint8_t r;
    uint8_t g;
    uint8_t b;
    uint8_t a;
} Pixel;

void applyGrayscale(uint8_t* img_ptr, int w, int h) {
    Pixel* img = (Pixel*)img_ptr; // 強轉型！
    int total = w * h;
    for (int i = 0; i < total; i++) {
        uint32_t gray = (img[i].r * 77 + ...) >> 8;
        img[i].r = img[i].g = img[i].b = gray;
    }
}
記憶體中的排布與對齊方式 (4-Byte 對齊)
R (0)
G (1)
B (2)
A (3)
R (4)
G (5)
B (6)
A (7)

WebAssembly 與 JS 的數據傳遞橋梁

線性記憶體模型 (Linear Memory)

Wasm 與 JS 無法直接互傳物件，數據的交換必須透過一塊被雙方共享的巨大記憶體緩衝區（ArrayBuffer）來進行。

  • HEAPU8 視圖：JS 透過 wasmModule.HEAPU8 存取該記憶體。
  • 指針的本質：在 JS 中，C 傳回的指針僅僅是一個代表記憶體偏移量 (Offset) 的整數 (Number)。
1
分配: JS 呼叫 _allocMemory(size) 在 Wasm 堆疊上取得空間位址。
2
寫入: JS 使用 HEAPU8.set(canvasPixels, srcPtr) 拷貝像素資料。
3
執行: 呼叫 C 導出的 Wasm 函數（如 _cropImage、_rotateImage）。
4
讀取: JS 藉由 HEAPU8.subarray(dstPtr, dstPtr + dstSize) 取得處理結果。

工程挑戰：手動記憶體管理與零拷貝讀取

// 批次處理的安全生命週期設計
let srcPtr = null;
let dstPtr = null;
try {
    srcPtr = wasmModule._allocMemory(srcSize);
    wasmModule.HEAPU8.set(pixels, srcPtr);
    
    // 執行運算...
    
    // 零拷貝(Zero-copy)讀取：subarray 僅映射不拷貝
    result = wasmModule.HEAPU8.subarray(dstPtr, dstPtr + dstSize);
} finally {
    // 核心防禦：確保釋放 Wasm 記憶體，防止洩漏崩潰！
    if (srcPtr) wasmModule._freeMemory(srcPtr);
    if (dstPtr) wasmModule._freeMemory(dstPtr);
}

⚠️ 批次處理的「防溢出」重點

記憶體洩漏風險：Wasm 分配的記憶體不受 JS 垃圾回收 (GC) 的控管。在批次處理數十或數百張大圖時，如果有一張圖處理完沒有調用 _freeMemory，記憶體用量將會線性暴增，直接導致瀏覽器頁面閃退崩潰。

Wasm 邊界零拷貝：HEAPU8.subarray() 僅在現有 ArrayBuffer 上建立視圖而不進行資料拷貝，實現了 Wasm 到 JS 邊界的零拷貝讀取。但在畫回 Canvas 時，因 ImageData 要求的 Uint8ClampedArray 型別限制，且為防範 Wasm 記憶體增長（Memory Growth）導致引用失效，JS 仍須執行一次深拷貝。

AI Agent 開發範式：自動化閉環測試實踐

如何有效指揮 AI Agent 實作？

  • 精確邊界規格：將 C 側定義為純計算函數；JS 側只處理瀏覽器 API、I/O 與打包，這能讓 AI Agent 對任務的職責一目了然。
  • 自主測試工具組：提供 pnpm dlx serve 讓 Agent 可以自主運行本地伺服器；利用 chrome-devtools-mcp 賦予 Agent 控制瀏覽器與進行行為模擬的能力。

🔄 自動自我修復閉環 (Feedback Loop)

在開發本專案的過程中，Agent 透過自動化工具開啟 Chrome 模擬操作，上傳測試圖檔並觸發 Wasm 計算。

Agent 自主檢測到 Wasm 控制台的「多張圖片寬高比不一致會導致裁切偏移」問題後，無需人類指示，即刻在 index.js 中補上了自動計算寬高比、浮點數防呆與出錯提示的防禦邏輯，實現了「編寫 -> 測試 -> 修正 -> 交付」的零人工干預閉環開發。
